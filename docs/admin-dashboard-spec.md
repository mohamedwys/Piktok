### Honest assessment: extend or rewrite?

**Extend.** The existing structure is already exactly right:

- Auth gate is solid and re-used by every privileged action (defense-in-depth pattern explicit at [web/src/lib/admin/auth.ts:16-23](web/src/lib/admin/auth.ts)).
- Service-role client is properly scoped, comment-documented, and dual-checked at compile and runtime.
- Server Components + GET-form filtering matches the existing web codebase conventions perfectly (no TanStack Query installed; no SWR; no client-side data layer).
- i18n catalog already has an `admin` namespace with translated strings.

The only structural change needed is **moving the existing `/admin` page (subscription list) to `/admin/subscriptions`** and freeing up `/admin` to be the new ops landing page. The detail page already lives at `/admin/subscriptions/<id>`, so the move is a single-file rename + a redirect.

## 3. Proposed dashboard layout

### Charting library decision

**Recommendation: skip charting libraries in v1.** The `web/package.json` has no chart dep installed today (`lucide-react`, `react-markdown`, `next`, `next-intl`, `stripe`, `@supabase/*` — no recharts/chartjs/d3). The existing admin patterns use server-rendered tables and KPI tiles, which serve the operator goal cleanly without adding ~80kB of client JS.

If charts become genuinely necessary later (e.g. signup funnel over time), the cleanest add is **`recharts` (~50kB tree-shakable, official React, Apache-2.0)** — flagged as an Open Question, not a Phase 2 decision.

### Server vs client components

**All Server Components, force-dynamic, GET-form filtering, Server Actions for mutations.** This matches the existing admin code byte-for-byte:

- Pages: `export const dynamic = 'force-dynamic'` + async Server Component.
- Filters: plain `<form method="get">` with `<input name="q">`, `<select name="status">`. Server re-renders.
- Mutations: Server Actions (Next 15 stable) — `'use server'` functions called from forms / button-only Client Components. Falls back to API routes for actions that need the Stripe SDK.

**No TanStack Query.** Web doesn't have it; we don't need it. This is `force-dynamic` SSR with GET-form params and `revalidatePath()` after Server Actions.

## 4. Moderation actions

### General principles

1. **Every moderation action is logged to `admin_actions` (§5).** No exceptions.
2. **Service-role only.** The web admin client (`getSupabaseAdmin()`) is the only path; mobile cannot perform any of these.
3. **Reversibility column-level.** Soft-state is a nullable timestamp + nullable reason text. Setting clears, NULL-ing un-sets. Lets us preserve the audit trail (when, why) without proliferating boolean flags.
4. **No new mobile reads required.** Mobile already tolerates listings disappearing (RLS filter, deleted product) — see §4.3 RLS impact below.

### 4.1 Suspend user (reversible) — `sellers.suspended_at`

Soft-state on the seller row. The user can still log in and read content, but cannot create/edit/delete listings, send messages, post comments, or follow.

**Schema change:**

```sql
alter table public.sellers
  add column if not exists suspended_at timestamptz,
  add column if not exists suspended_reason text;

create index if not exists sellers_suspended_at_idx
  on public.sellers (suspended_at)
  where suspended_at is not null;
```

**Mobile-app-impact check:**

- **Tables touched:** `public.sellers` (additive columns only).
- **Mobile callsites that read/write `sellers`:**
  - [src/features/marketplace/services/sellers.ts:82](src/features/marketplace/services/sellers.ts) — `getSellerById` `select('*')` ← will receive 2 new columns; ignored client-side.
  - [src/features/marketplace/services/sellers.ts:108](src/features/marketplace/services/sellers.ts) — `getMySeller` `select('*')` ← same.
  - [src/features/marketplace/services/sellers.ts:162](src/features/marketplace/services/sellers.ts) — `updateMySeller` `select('*')` after update ← same.
  - [src/features/marketplace/services/products.ts:246](src/features/marketplace/services/products.ts) — embedded seller `select('*')` after `searchNearby` ← same.
  - Several explicit-column reads (messaging, follows) — ignore the new columns automatically.
- **Risk to mobile app: none for column add alone.** `select('*')` returns extra fields the mobile types don't know about; TS will not error at runtime, and the regenerated `src/types/supabase.ts` will gain the fields after `npm run gen:types`.
- **Type regen required on mobile after Phase 2 lands** — `src/types/supabase.ts` must be regenerated for the new fields to appear in TS. Not blocking; existing reads keep working.
- **Rollback DDL:**
  ```sql
  begin;
    drop index if exists public.sellers_suspended_at_idx;
    alter table public.sellers
      drop column if exists suspended_at,
      drop column if exists suspended_reason;
  commit;
  ```

**RLS implication — how does suspension actually prevent activity?**

Adding the column alone doesn't enforce anything. We have two options to make it bite. Either we **modify existing INSERT/UPDATE policies** to add `... and (sellers.suspended_at is null)`, or we **add new policies that AND a suspension predicate** (Postgres RLS evaluates with OR semantics across multiple permissive policies for the same action — adding a permissive policy is additive, not restrictive). For a **restrictive** suspension gate, the cleanest path is to **edit the existing policies in place** to AND the suspended check.

Existing write policies that need a suspension predicate (one-line addition each):

| Policy (file:line) | Today's check | New check |
|---|---|---|
| `products insert own` ([20260503_sell_setup.sql:109-115](supabase/migrations/20260503_sell_setup.sql)) | `seller_id in (select id from sellers where user_id = auth.uid())` | + `and not exists (select 1 from sellers where user_id = auth.uid() and suspended_at is not null)` |
| `products update own` ([20260507_owner_update.sql:170-179](supabase/migrations/20260507_owner_update.sql)) | same shape | same addition |
| `products delete own` ([20260506_owner_delete.sql:152-159](supabase/migrations/20260506_owner_delete.sql)) | same shape | same addition |
| `messages insert own` ([20260509_messaging.sql:247-254](supabase/migrations/20260509_messaging.sql)) | sender_id = auth.uid() and conversation_id in (...) | + `and not exists (select 1 from sellers where user_id = auth.uid() and suspended_at is not null)` |
| `comments self insert` ([20260520_comments_schema.sql:267-276](supabase/migrations/20260520_comments_schema.sql)) | author_id in (own seller) | same addition |
| `follows self insert` ([20260518_follows_schema_and_counters.sql:218-227](supabase/migrations/20260518_follows_schema_and_counters.sql)) | follower_id in (own seller) | same addition |
| `likes insert own` ([20260501_initial_marketplace_schema.sql:95](supabase/migrations/20260501_initial_marketplace_schema.sql)) | auth.uid() = user_id | same addition |
| `bookmarks insert own` ([20260501_initial_marketplace_schema.sql:99](supabase/migrations/20260501_initial_marketplace_schema.sql)) | auth.uid() = user_id | same addition |

**This IS a tightening of existing RLS policies. Flagged.** A non-suspended user is unaffected; a suspended user gets `403` from Postgres on any of the above writes.

**Mobile error tolerance for the new 403 path:** mobile already throws on RLS rejections (the audit confirms list queries handle empty `data ?? []`, but write rejections surface as Postgres errors that bubble up to the UI). The user-visible consequence is a generic error toast on attempted post/message/comment/like — acceptable for a suspended user. **No mobile code change needed**, but Phase 2 commit 3 must include a mobile smoke test confirming the *non-suspended* path still works (see §7).

**Audit-log entry:** `admin_actions(action_type='suspend_user', target_type='seller', target_id=<seller.id>, reason=<text>, metadata={'suspended_at':...})`. Unsuspend is `action_type='unsuspend_user'` with the same target shape.

**API surface (Server Actions — no separate route):**

- `suspendUser(sellerId: string, reason: string)` → service-role `update sellers set suspended_at=now(), suspended_reason=reason where id=$1`. Then write `admin_actions`. Then `revalidatePath('/admin')`.
- `unsuspendUser(sellerId: string)` → set both columns NULL, write `admin_actions(action_type='unsuspend_user')`.

### 4.2 Ban user (permanent) — `auth.users.banned_until`

Hard ban via Supabase's built-in mechanism. **No schema change.** The mobile login flow rejects with `400 User is banned` from `auth.signInWithPassword`.

**API:**

```ts
const admin = getSupabaseAdmin();
await admin.auth.admin.updateUserById(authUserId, {
  ban_duration: 'none', // or '8760h' for 1 year, etc.
});
```

To unban: `ban_duration: 'none'` reverses (Supabase docs).

**Mobile-app-impact check:**

- **Tables touched:** `auth.users` only (managed table; we use the admin API, not direct DDL).
- **Mobile callsites:** [src/features/auth/services/auth.ts](src/features/auth/services/auth.ts) reads `getUser()` and uses `signInWithPassword` (verified via grep). On ban, the next sign-in attempt fails — **expected and intended**. Already-signed-in sessions stay valid until token expiry (~1h); the user can still navigate the cached app shell until the access token is rejected on the next request.
- **Risk to mobile app: none for non-banned users.** Banned users are intentionally locked out; the mobile sign-in screen surfaces the standard error path.
- **No type regen needed** (no schema change).
- **No rollback DDL** (use `ban_duration: 'none'` to reverse).

**Audit-log entry:** `admin_actions(action_type='ban_user', target_type='auth_user', target_id=<auth.users.id>, reason=<text>, metadata={'ban_duration': '...'})`.

### 4.3 Hide listing (soft, public-invisible but owner can see) — `products.hidden_at`

Soft-removal. The listing disappears from the public feed but stays in the owner's "My listings" surface so they can re-edit / understand why it was hidden (out of v1 scope: surfacing the reason to the user, but storing it now keeps the door open).

**Schema change:**

```sql
alter table public.products
  add column if not exists hidden_at timestamptz,
  add column if not exists hidden_reason text;

create index if not exists products_hidden_at_idx
  on public.products (hidden_at)
  where hidden_at is not null;
```

**Mobile-app-impact check:**

- **Tables touched:** `public.products` (additive columns only).
- **Mobile callsites that read `products`:** all SELECT paths use `select('*')` plus an embedded `seller:sellers(*)` (per the audit) — every one will receive the new columns and ignore them.
- **`featured_until` precedent** ([20260524_featured_listings.sql](supabase/migrations/20260524_featured_listings.sql)) already established the same pattern of adding a nullable `timestamptz` column to `products` for system state — the mobile app handled that addition transparently. Same shape, same risk profile.
- **Risk to mobile app: low.** The non-trivial part is the RLS change below, not the column add.
- **Type regen required on mobile after Phase 2 lands.**
- **Rollback DDL:**
  ```sql
  begin;
    drop index if exists public.products_hidden_at_idx;
    alter table public.products
      drop column if exists hidden_at,
      drop column if exists hidden_reason;
  commit;
  ```

**RLS implication — making "hidden" actually hide:**

The current `products public read` policy is `using (true)` — anyone can read every row. To hide, we **edit it** to `using (hidden_at is null)`, and **add a new permissive policy** so the owner still sees their own hidden listings:

```sql
-- Tighten: hide hidden_at-set rows from public reads
drop policy if exists "products public read" on public.products;
create policy "products public read"
  on public.products
  for select
  using (hidden_at is null);

-- New permissive policy: owner always sees their own listings
create policy "products owner read own"
  on public.products
  for select
  using (
    seller_id in (select id from public.sellers where user_id = auth.uid())
  );
```

**This IS a tightening of an existing RLS policy. Flagged.**

- **Risk to mobile app:** the mobile feed (`listProducts`, `searchProducts`, `searchNearby` via `products_within_radius` RPC, `listFeaturedProducts`, related products) will stop returning rows where `hidden_at is not null`. That is the *intent*. Owner-side surfaces (My listings, listing detail for own product) keep working via the new `products owner read own` policy.
- **Anonymous (`anon`) users** will see exactly the public catalog minus hidden — same as today minus the hidden ones.
- **The `products_within_radius` RPC** is `security invoker`, so the RLS change applies to it transparently — no RPC edit needed.

**Audit-log entry:** `admin_actions(action_type='hide_listing', target_type='product', target_id=<product.id>, reason=<text>, metadata={'hidden_at':...})`.

### 4.4 Delete listing (hard) — DELETE on `products`

Already supported via service-role `delete from products where id=$1`. Cascades clean up likes, bookmarks, comments, follows-like rows (none point to products, but `product_views` and `bookmarks/likes/comments` cascade per their FKs).

**Mobile-app-impact check:**

- No schema change.
- Mobile reads: any cached references to the deleted listing fail to re-fetch (returns 0 rows). Mobile tolerates this — `getProductById` returns null and the UI treats it as not-found per the existing handling.
- Conversations referencing the deleted product **cascade-delete** ([20260509_messaging.sql:194-201](supabase/migrations/20260509_messaging.sql)) — the buyer's chat list loses the row. Mobile `listConversations` returns the smaller set. Acceptable.
- Orders referencing the deleted product would BLOCK on `RESTRICT` ([20260510_orders.sql:331](supabase/migrations/20260510_orders.sql)) — admin must NOT delete a listing with paid orders. **Phase 2 implementation must check this**: query `select count(*) from orders where product_id = $1` first; refuse delete if non-zero, surface a "Use Hide instead" UX hint.

**Audit-log entry:** `admin_actions(action_type='delete_listing', target_type='product', target_id=<product.id>, reason=<text>, metadata={'snapshot': <full row jsonb>})` — store a JSON snapshot in `metadata` so the audit log is forensically complete after the row is gone.

### 4.5 Resolve report — new `reports` table

There is no `reports` table today. Mobile has a stub "Report" button at [src/components/feed/MoreActionsSheet.tsx:74-76](src/components/feed/MoreActionsSheet.tsx) that shows a "Coming soon" alert.

For the dashboard's report queue to be meaningful in Phase 2, we either:

(a) **Add the table now and seed it with synthetic rows** for QA, leaving the mobile wire-up for Phase 3+.
(b) **Add the table now AND wire up the mobile button** in the same release.

Recommend **(a)** for Phase 2 — keeps the mobile change off the critical path of the admin rebuild. The mobile wire-up becomes an independent PR later.

**Schema change (new table):**

```sql
create table if not exists public.reports (
  id uuid primary key default uuid_generate_v4(),
  reporter_id uuid not null references auth.users(id) on delete set null,
  target_type text not null check (target_type in ('product', 'seller', 'comment', 'message')),
  target_id uuid not null,
  reason_code text not null check (reason_code in ('spam','nsfw','scam','harassment','illegal','other')),
  reason_text text,
  status text not null default 'open'
    check (status in ('open','resolved','escalated','dismissed')),
  resolved_by uuid references auth.users(id) on delete set null,
  resolved_at timestamptz,
  resolution_note text,
  created_at timestamptz not null default now()

create index reports_status_idx on public.reports(status) where status = 'open';
create index reports_target_idx on public.reports(target_type, target_id);
create index reports_reporter_idx on public.reports(reporter_id);

alter table public.reports enable row level security;

-- Reporter can insert their own report (Phase 3+ wire-up)
create policy "reports self insert"
  on public.reports
  for insert
  to authenticated
  with check (reporter_id = auth.uid());

-- No SELECT/UPDATE/DELETE policies for authenticated/anon — admin-only via service-role
```

**Mobile-app-impact check:**

- **Tables touched:** new `public.reports` only — additive.
- **Mobile callsites:** none today (the "Report" button is a stub). The audit confirms no `reports`, `is_suspended`, `suspended_at`, `banned_at`, `is_hidden`, `hidden_at` references in the mobile tree.
- **Risk to mobile app: none.** Adding a new table is invisible until the mobile codebase chooses to read/write it.
- **Type regen optional on mobile** until the mobile wires up the button.
- **Rollback DDL:**
  ```sql

    drop policy if exists "reports self insert" on public.reports;
    drop index if exists public.reports_reporter_idx;
    drop index if exists public.reports_target_idx;
    drop index if exists public.reports_status_idx;
    drop table if exists public.reports;
  ```

## 6. Auth & access control

### Identity choice — admin role storage

**Pick (a) `is_admin` boolean on `sellers`.** Justification:

1. **Already in place** ([20260523_add_is_admin_to_sellers.sql](supabase/migrations/20260523_add_is_admin_to_sellers.sql)). Switching now would be code churn for no benefit.
2. **Already enforced** by [web/src/lib/admin/auth.ts:47-55](web/src/lib/admin/auth.ts).
3. **Single join** in the gate (one `select` against the `sellers` row that's already keyed by user). Option (c) (separate `admins` table) adds a join with no v1 win.
4. **`auth.users.app_metadata.role`** (option b) requires service-role to set and is harder to migrate later if we want multi-role. Migrating from `is_admin` to a `role` column on `sellers` (or a separate `admins(user_id, role)` table) later is a copy-and-drop, mechanical.

**Mobile-app implication of using `is_admin` on `sellers`:**

- Mobile DOES read `sellers` via `select('*')` — see the audit. `is_admin` is already in the row shape (added by 20260523).
- Mobile does NOT read `is_admin` anywhere (audit verified).
- Mobile does NOT read `auth.users.app_metadata.role` anywhere (audit verified).
- Adding admin fields to `sellers` is invisible to mobile in practice.

### Middleware / layout guard for `/admin/*`

Today's middleware ([web/src/middleware.ts](web/src/middleware.ts)) only does locale routing + Supabase session refresh. Auth gating stays at the **page level** via `requireAdmin()` — that's the existing convention and the docstring at [web/src/middleware.ts:30-32](web/src/middleware.ts) explicitly calls it out.

**Recommendation:** add a route-group `layout.tsx` at `web/src/app/[locale]/admin/layout.tsx` that calls `requireAdmin(locale)` once at the layout level. Every page under `/admin/*` is then gated transparently. Per-page `requireAdmin()` calls stay (defense-in-depth) and per-API-route `requireAdminApi()` calls stay (defense-in-depth).

```ts
// web/src/app/[locale]/admin/layout.tsx
export default async function AdminLayout({
  children, params,
}: { children: React.ReactNode; params: Promise<{ locale: string }> }) {
  const { locale } = await params;
  setRequestLocale(locale);
  await requireAdmin(locale);
  return <>{children}</>;
}
```

### Bootstrap: make m.w.lamane@gmail.com the first admin

The seller row only exists once the user has signed in to the mobile app or web (it's lazily created by `get_or_create_seller_for_current_user` RPC). So the bootstrap is two-step:

1. **One-time mobile login** — m.w.lamane@gmail.com signs into the mobile app once. This creates the `auth.users` row AND, on first profile-touching action, the `sellers` row.
2. **Run this SQL in Supabase SQL editor:**

```sql
update public.sellers
   set is_admin = true
 where user_id = (
   select id from auth.users where email = 'm.w.lamane@gmail.com'
 );
```

Verify with `select id, name, is_admin from public.sellers where is_admin = true;`. The partial index `sellers_is_admin_idx` keeps this query free.

**If the seller row doesn't exist yet** (user signed up but never finished the profile flow), the SQL update affects 0 rows. In that case, manually insert a stub:

```sql
insert into public.sellers (user_id, name, is_admin)
select id, 'Admin', true from auth.users where email = 'm.w.lamane@gmail.com'
on conflict (user_id) do update set is_admin = true;
```

### Service-role client usage strategy

- **Server-side only.** The web admin client lives at [web/src/lib/supabase/admin.ts](web/src/lib/supabase/admin.ts) and is consumed exclusively by Server Components and Route Handlers under `web/src/app/`. Never imported into a Client Component (`'use client'`).
- **Module-level cached** singleton (already implemented).
- **Env var:** `SUPABASE_SERVICE_ROLE_KEY` is server-only. **Never** prefixed `NEXT_PUBLIC_` — that would inline it into the client bundle. Today's [admin.ts:15-23](web/src/lib/supabase/admin.ts) docstring already calls this out.
- **Mobile uses anon key** (per `app.json` and [src/lib/supabase/client.ts] equivalent). The service role must NEVER appear in any mobile bundle, EAS build secret, or app.json.
- **Vercel env:** `SUPABASE_SERVICE_ROLE_KEY` set in Production / Preview / Development scopes individually. `eas.json` (mobile) must NOT contain this var — verify before Phase 2 ships. Per the memory note about EAS env vars: grep the mobile build artifacts post-deploy to confirm `SUPABASE_SERVICE_ROLE_KEY` does not appear anywhere in `dist/` or the iOS/Android bundle.
